# EstateCRM — Генератор email-подписей

Внутренний многопользовательский сервис для создания фирменных подписей сотрудников.
Работает полностью статически на GitHub Pages.

**Сервис:** https://andrewbalanov.github.io/estatecrm-signatures/

## Возможности

- **Приглашения**: администратор приглашает сотрудника (имя, фамилия, email, отдел,
  должность, мобильный, роль) — сервис генерирует личный код доступа и готовое письмо-приглашение.
- **Роли**: администратор видит и редактирует всех, приглашает/удаляет сотрудников,
  создаёт шаблоны. Пользователь видит только свою подпись, редактирует свои данные и фото.
- **Шаблоны**: конструктор для администратора — название, приветствие, телефон, сайт, цвета,
  кнопка видеозвонка, соцсети (иконки + ссылки) и **обязательные поля** (включая фото):
  пока сотрудник их не заполнит, кнопки установки подписи неактивны.
- **Установка**: копирование, оптимизированное под Outlook Windows, Outlook Mac,
  Mail Mac, Mail iPhone — с пошаговыми инструкциями.

## Безопасность (конвертное шифрование)

| Файл | Что хранит |
|---|---|
| `data/users.json` | Для каждого пользователя: email, роль, соль, верификатор кода (PBKDF2-SHA256, 600 000 итераций) и **конверт** — общий ключ данных, зашифрованный личным ключом пользователя (AES-256-GCM). Плюс `encToken` — GitHub-токен, зашифрованный ключом данных. |
| `data/employees.json.enc` | Данные сотрудников, зашифрованы общим ключом данных. |
| `data/templates.json` | Настройки шаблонов (открытые — они и так видны в каждой подписи). |
| `assets/photos/` | Фотографии (публичные — их загружают почтовые программы получателей). |

Вход = email + личный код доступа. Код выводит личный ключ, тот открывает конверт с общим
ключом данных, а он — базу сотрудников и GitHub-токен для сохранения. Посторонний видит в
репозитории только соли, верификаторы и шифртексты.

**Модель доверия (важно):** все вошедшие сотрудники получают общий ключ данных и токен записи —
технически любой сотрудник может прочитать данные коллег и писать в репозиторий. Роли
разграничивают интерфейс, а не криптографию. Для внутреннего инструмента компании это осознанный
компромисс. При увольнении сотрудника: удалите его в сервисе **и** перевыпустите GitHub-токен
(⚙ в шапке) — старый мог остаться у него в браузере.

## Приглашение сотрудника

«+ Пригласить сотрудника» → заполнить поля (email для входа может отличаться от email в
подписях), отметить, какие подписи создать, выбрать роль → «Сохранить». Сервис сгенерирует
**персональную одноразовую ссылку** и текст приглашения (для администратора — свой вариант
текста): скопируйте его сотруднику в чат или отправьте по email. Открыв ссылку, сотрудник
сам придумывает пароль; после этого ссылка перестаёт действовать. Потерянный пароль —
«Новая ссылка для входа» в карточке сотрудника.

У сотрудника может быть несколько подписей (по разным шаблонам), у каждой — свой email.

## Изменение макета шаблонов

Структура и вёрстка подписи (email-safe HTML) — в `js/templates.js`, меняется через Claude.
Всё остальное (поля, цвета, соцсети, обязательность) — в интерфейсе, вкладка «Шаблоны».

## Служебные скрипты

```bash
# Прочитать базу сотрудников (любой действующий доступ):
SIG_EMAIL='email' SIG_PASSWORD='код' node tools/decrypt-data.mjs

# Полная переинициализация (сбрасывает ВСЕ коды доступа):
SIG_EMAIL='email админа' SIG_PASSWORD='код' node tools/init-data.mjs seed.json

# Разовая миграция v1 → v2 (уже выполнена):
SIG_PASSWORD='пароль' node tools/migrate-v2.mjs
```

## Режимы

- Демо: добавьте `#dev` к адресу — изменения живут только в памяти браузера.
- «Только просмотр»: администратор ещё не подключил GitHub-токен (⚙ в шапке).
